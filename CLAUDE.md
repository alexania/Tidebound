# Tidebound

## What this is

A single-player browser murder mystery puzzle game. Action-driven. No luck. Coastal
setting — culture, era, tone, and genre vary per scenario. Scenarios generated at
runtime via LLM (Anthropic Claude).

The design is being actively iterated. Treat any design conversation as a brainstorm
where you have a vote — not a yes.

---

## Hard Rules (not up for discussion)

These are bedded in. Don't suggest changes without a compelling reason.

1. **Platform**: Browser game. React + TypeScript + Vite. Plain CSS. No backend.
2. **Player role**: The investigator. The player moves through the world, inspects
   locations and items, and talks to characters. They are present in the world.
3. **Core game objects**: A scenario consists of a crime, locations (grid map),
   characters, items, clues, and checkpoints.
4. **Checkpoint gating**: Investigative checkpoints first (cause of death, murder
   location, time of death). Only once all 3 are confirmed does the accusation phase
   unlock (perpetrator → motive).
5. **LLM generation**: Scenarios are generated by Claude at runtime if no unplayed one
   is available. We provide a system prompt + JSON schema. Diversity is a goal;
   repetition is failure.
6. **Action-driven**: Clues fire immediately on player actions (inspect, talk, ask).
   No turns. No batched resolution.

---

## Core Design Principles

These govern every design decision. When an idea conflicts with one, say so.

### 1. Logically solvable
This is a puzzle, not a lottery. Every correct answer must be reachable through clues
alone. A player who reads every clue carefully should be able to solve it. Red herrings
are allowed — confusion is allowed — but the correct path must exist and be sufficient.

**Implication**: Clue design must be traceable. If a player solves a checkpoint, they
should be able to explain *why* that answer is correct from clues they collected.

### 2. Fun
Atmosphere, narrative, and pacing matter. The player should feel clever when they solve
something, not relieved when they guess right.

### 3. Clear first step
When a new game starts, the player must understand what to do. "What now?" is a design
failure. Leads are surfaced on arrival pointing toward specific characters and locations.

## Current Mechanics

### Player actions
Five actions, each fires relevant clues immediately:

- **Move** to an adjacent location — passive, reveals NPCs present, no clues fire
- **Inspect location** — fires `investigator_at_location` clues; reveals visible items
  via `location_discovery_text`; also fires `investigator_at_location_with_item` for
  items currently in inventory
- **Inspect item** — picks up item into inventory if not already held; fires
  `investigator_with_item` clues; fires `investigator_at_location_with_item` clues
- **Talk to character** — fires `investigator_with_character` clues
- **Ask character about item** (must be holding item) — fires
  `investigator_with_character_and_item` clues

### Clue conditions
5 condition types — all involve the investigator:
`investigator_at_location` · `investigator_with_character` · `investigator_with_item` ·
`investigator_at_location_with_item` · `investigator_with_character_and_item`

See `src/engine/conditions.ts`.

### Clue firing
Clues fire immediately when the matching action is taken. Each clue fires once.
Accusation clues (`perpetrator`, `motive`) are gated until all investigative
checkpoints are confirmed; attempting a gated action shows a feedback string instead.

### Items
Items start at a `starting_location`. They are invisible until the player inspects that
location (`location_discovery_text` reveals them). Inspecting an item picks it up into
inventory. Items are carried — there is no map presence for items.

### Evidence board
Players pin clues as cards and add notes. Cards are dragged into checkpoint swim lanes —
one lane per checkpoint. Submitting a checkpoint answer is done directly from the lane
header via a dropdown; cards in that lane are auto-cited.

### Scoring
`max(base - actionCount * 15, 100)`. Base: easy 1000 / medium 2000 / hard 3000.

### Feedback strings
Static strings in `src/engine/feedback.ts`, keyed by action type and result:
- **empty**: no matching clue exists for this action
- **locked**: a clue exists but accusation gate is closed
- **missing** (easy only): a clue exists but requires an item not yet in inventory

## Tech Architecture

### Key pattern
Engine (`src/engine/`) is pure functions only — no React, no side effects. All game
state lives in `App.tsx`, passed down as props. Components call engine functions; they
don't contain game logic.

### LLM generation
`src/engine/generator.ts` — calls Anthropic API directly from the browser.
Model: `claude-opus-4-6`. Validates against strict schema, retries up to 3 times.
Player supplies their own API key. Genre, tone, and culture are seeded randomly per
generation for variability. The only hard setting constraint is coastal.

### Locations
Up to 9 locations per scenario, placed in a 3×3 grid by `col` and `row`. Location IDs
are scenario-specific (not fixed globally). Adjacency is defined explicitly per scenario
via `location_adjacencies`.

### Tag format
LLM output uses `[char:Name]`, `[loc:location_id]`, `[item:Name]`, and `[time:phrase]`
throughout narrative text. Parsed by `src/utils/parseTags.tsx` for UI rendering.
`location_discovery_text` on items uses the same tag format.

## Design Discussion Mode

The game concept is actively being shaped. When the user proposes a design change:

- **Think it through before agreeing.** Does it conflict with a core principle? Say so.
- **Have an opinion.** Don't validate whatever was said last.
- **Push back on drift.** If an idea moves away from "logically solvable" or "clear
  first step", name it explicitly.
- **Distinguish iteration from redesign.** Changes to the core loop (action-based clue
  firing, condition types, checkpoint gating) are significant — treat them as such.
- **Suggest alternatives** when rejecting something. "No" alone isn't useful.

Design questions are brainstorms, not instructions.
