# Tidebound

## What this is

A single-player browser murder mystery puzzle game. Turn-based. No luck. Folk horror
atmosphere. Scenarios generated at runtime via LLM (Anthropic Claude).

The design is being actively iterated. Treat any design conversation as a brainstorm
where you have a vote — not a yes.

---

## Hard Rules (not up for discussion)

These are bedded in. Don't suggest changes without a compelling reason.

1. **Platform**: Browser game. React + TypeScript + Vite. Plain CSS. No backend.
2. **Player role**: Omniscient situation manipulator. The player is not present in the
   world. They arrange characters and items; they don't move themselves or speak.
3. **Core game objects**: A scenario consists of a crime, locations (fixed 9-grid),
   characters, items, clues, and checkpoints.
4. **Checkpoint gating**: Investigative checkpoints first (cause of death, murder
   location, time of death, victim state). Only once all 4 are confirmed does the
   accusation phase unlock (perpetrator → motive → hidden_truth on hard).
5. **LLM generation**: Scenarios are generated by Claude at runtime if no unplayed one
   is available. We provide a system prompt + JSON schema. Diversity is a goal;
   repetition is failure.
6. **Turn-based**: Each turn the player has 3 actions to move characters/items. At end
   of turn, clues fire based on board state conditions. The player cannot move a clue
   into existence — they create the conditions for it.

## Core Design Principles

These govern every design decision. When an idea conflicts with one, say so.

### 1. Logically solvable
This is a puzzle, not a lottery. Every correct answer must be reachable through clues
alone. A player who reads every clue carefully should be able to solve it. Red herrings
are allowed — confusion is allowed — but the correct path must exist and be sufficient.

**Implication**: Clue design must be traceable. If a player solves a checkpoint, they
should be able to explain *why* that answer is correct from clues they collected.

### 2. Fun
Atmosphere, narrative, and pacing matter. The player should feel clever when they solve
something, not relieved when they guess right. The LLM tone (folk horror, restrained,
eerie) serves this goal — don't erode it.

### 3. Clear first step
When a new game starts, the player must understand what to do. "What now?" is a design
failure. Currently: leads are surfaced on turn 1 pointing toward specific conditions.
This is the current answer — it may not be the final one.

## Current Mechanics

### Turn structure
- **Setup phase**: 3 actions to move characters and/or items
- **Resolve**: Engine fires clues whose conditions match current board state
- **Review**: Player submits checkpoint answers directly from the evidence board

### Clue conditions
8 condition types control when a clue fires — combinations of which characters are at
which location, and whether items are present. See `src/engine/conditions.ts`.

### Clue firing
All clues whose conditions are met fire each turn — no cap per location.

### Items
Items start at a `starting_location`. Hidden items have a `discovery_condition` and only
appear once that condition is met. Visible items are immediately available to move.

### Evidence board
Players pin clues as cards and add notes. Cards are dragged into checkpoint swim lanes —
one lane per checkpoint. Submitting a checkpoint answer is done directly from the lane
header via a dropdown; cards in that lane are auto-cited. Hard clues are auto-pinned to
the Uncategorised lane on collection.

### Scoring
`max(base - turn * 50, 100)`. Base: easy 1000 / medium 2000 / hard 3000.

## Tech Architecture

### Key pattern
Engine (`src/engine/`) is pure functions only — no React, no side effects. All game
state lives in `App.tsx`, passed down as props. Components call engine functions; they
don't contain game logic.

### LLM generation
`src/engine/generator.ts` — calls Anthropic API directly from the browser.
Model: `claude-opus-4-6`. Validates against strict schema, retries up to 3 times.
Player supplies their own API key.

### Locations
Fixed 9 IDs: `harbour`, `tavern`, `lighthouse`, `chapel`, `doctors_house`, `manor`,
`cottage_row`, `cliffs`, `forest_edge`. These are the permanent map. Not configurable.

### Tag format
LLM output uses `[char:Name]`, `[loc:location_id]`, `[item:Name]`, and `[time:phrase]`
throughout narrative text. Parsed by `src/utils/parseTags.tsx` for UI rendering.

## Design Discussion Mode

The game concept is actively being shaped. When the user proposes a design change:

- **Think it through before agreeing.** Does it conflict with a core principle? Say so.
- **Have an opinion.** Don't validate whatever was said last.
- **Push back on drift.** If an idea moves away from "logically solvable" or "clear
  first step", name it explicitly.
- **Distinguish iteration from redesign.** Changes to the core loop (3 actions,
  condition-based clues, checkpoint gating) are significant — treat them as such.
- **Suggest alternatives** when rejecting something. "No" alone isn't useful.

Design questions are brainstorms, not instructions.
