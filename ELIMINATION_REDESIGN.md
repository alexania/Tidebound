# Elimination Mechanic Redesign

Replaces the convergence model (2 correct + 3 red herrings per checkpoint) with a pure
elimination model. All clues are true observations. The correct answer for each checkpoint
is the one not contradicted by any clue. Players prove each wrong answer impossible before
the correct answer is accepted.

---

## Design decisions (settled)

- **All clues are true.** No `weight` field. No red_herring_explanation.
- **Clues are not checkpoint-bound.** A single clue can contradict wrong answers across
  multiple checkpoints. No `checkpoint` or `answer` field on Clue.
- **Each clue has a `contradicts` array** listing which (checkpoint, answer) pairs it
  makes logically impossible.
- **Submission requires proof.** To submit a checkpoint answer, the player must have
  collected at least one clue that contradicts each wrong answer for that checkpoint.
  The UI makes this explicit: one slot per wrong answer, player assigns a clue to each.
  When all slots are filled, the remaining answer is auto-selected and submission unlocks.
- **One clue per wrong answer slot is sufficient**, even if more contradicting clues exist.
- **Difficulty = answer option count.** Easy: 3 options per checkpoint (2 wrong options to
  eliminate). Medium: 4–5. Hard: 5–6. Same clue set regardless of difficulty; harder
  means more wrong answers to eliminate.
- **Accusation gate moves to submission, not clue firing.** Perpetrator and motive clues
  fire freely at any time. But the submission UI for those two checkpoints is locked until
  all three investigative checkpoints (cause_of_death, true_location, time_of_death) are
  confirmed. Clues visible; submission blocked.
- **Existing bundled scenarios will not be migrated.** A new scenario will be generated
  under the new schema to validate the design. Old scenarios stay as-is and will break
  once the schema changes — they can be regenerated later.
- **Backward Compatibility can be ignored.** Since we're generating new scenarios we do not
  need to maintain any backward compatibility.

---

## Clue count

Scenarios are not generated by difficult, thus we always need to generate 20–28 clues and 5-6
options per checkpoint.

The engine needs to remove the extra options based on difficulty.
We could potentially remove clues that do not contradict any of the remaining options if we 
find that having "useless" clues in the game is annoying.

**Anti-clustering rule:** No single clue may appear in more than 3 `contradicts` entries
total across all checkpoints. Prevents the LLM from writing one vague clue that
"eliminates" everything and shortcuts the investigation.

**Coverage requirement:** For every checkpoint, for every wrong answer option, at least
one collected clue must contradict it. The validator enforces this.

**Clarity rule:** No clue may seem to contradict a correct answer. The LLM should double
check for this.

## Schema changes — `src/types/scenario.ts`

### Clue (full replacement)

Remove: `checkpoint`, `answer`, `weight`, `red_herring_explanation`, `requires_clue_id`

Add: `contradicts`

```typescript
interface Clue {
  id: string
  condition: ClueCondition
  text: string
  contradicts: Array<{ checkpoint: string; answer: string }>
}
```

`contradicts` entries must reference a valid checkpoint id and an answer string that
exactly matches an entry in that checkpoint's `answer_options`. Validator enforces both.

### Everything else unchanged
Checkpoint, Item, Location, Character, etc. schemas are unaffected.

---

## GameState changes — `src/types/gameState.ts`

### CheckpointState

Add a `proofs` map: which clue the player has assigned to contradict each wrong answer.

```typescript
interface CheckpointState {
  confirmed: boolean
  confirmedAnswer: string | null
  proofs: Record<string, string>  // wrong_answer → clue_id
}
```

`proofs` is populated as the player assigns clues in the evidence board UI. Submission
is enabled when every wrong answer for that checkpoint has an entry in `proofs` and the correct
answer does not have a clue assigned (and all
assigned clues are in the player's collected clues).

## Engine changes — `src/engine/gameEngine.ts`

- **Remove** all locked-clue logic (the accusation gate suppression on clue firing).
- **Remove** difficulty-based clue filtering on game init.
- **Add** difficulty-based option filtering on game init.
- **`submitCheckpoint`**: validate that for every wrong answer in that checkpoint's
  `answer_options`, `state.checkpoints[id].proofs` has a valid clue_id. If the checkpoint
  is `perpetrator` or `motive`, also validate that all three investigative checkpoints are
  confirmed. Return an error string if either validation fails.
- **`assignProof(checkpointId, wrongAnswer, clueId)`**: new action handler. Validates the
  clue is in collected clues and its `contradicts` array includes this checkpoint/answer
  pair. Updates `CheckpointState.proofs`.

---

## Validator changes — `src/engine/validator.ts`

Add checks:
1. Every `contradicts` entry references a valid `checkpoint.id`.
2. Every `contradicts` entry's `answer` exactly matches an option in that checkpoint's
   `answer_options` and is NOT the correct answer (can't contradict the right answer).
3. **Coverage**: for every checkpoint, for every wrong answer option (all options except
   the one matching `crime.*`), at least one clue's `contradicts` array covers it.
4. **Anti-clustering**: no clue has more than 3 `contradicts` entries total.
5. Clue count within the range for the scenario's difficulty.

Note: validator needs access to the `crime` block to know which answer is correct per
checkpoint (cause_of_death → crime.cause_of_death, etc.).

## Generator prompt rewrite — `src/engine/generator.ts`

### What to remove
- "2 correct + 3 red_herring per checkpoint" structure
- `weight` and `red_herring_explanation` fields
- Checkpoint-bound clue planning
- Difficulty stripping logic
- Accusation gate clue suppression language

### What to replace it with

**Clue planning (Phase 1, Step 4):**

> Plan clues as true observations — physical evidence, direct testimony, discoverable
> facts. Each clue has a `contradicts` array listing which wrong answers across which
> checkpoints it makes logically impossible.
>
> First list every wrong answer option for every checkpoint (the correct answers come
> from the crime block — every other option is a wrong answer that must be eliminated).
> Then plan clues sufficient to cover all of them. At least one clue must exist for each
> wrong answer. A single clue may contradict multiple wrong answers across multiple
> checkpoints, but no clue should appear in more than 3 contradicts entries — if a clue
> seems to eliminate everything, it is too vague; make it more specific.
>
> Aim for 20–28 clues.

**Consistency check (Phase 1, Step 5):**

> Build a coverage table: checkpoint × wrong answer → which clue(s) contradict it.
> Every cell must be filled. If any wrong answer has no contradicting clue, write one.
> Check that no clue is doing more than 3 contradictions — if so, split it or make it
> more specific.

**Answer options:**

> Answer options per checkpoint: 5–6. The correct answer is always included. Wrong answers
> should be plausible given the setting — not obviously wrong from the opening narrative alone.

**JSON schema update:** Remove `weight`, `checkpoint`, `answer`, `red_herring_explanation`
from Clue. Add `contradicts: [{checkpoint, answer}]`.

### Accusation gate language

> Perpetrator and motive clue conditions may fire at any time — do not treat them
> differently from investigative clues. Design their conditions to require deliberate
> routing (item interactions, specific characters) so they are naturally discovered after
> the investigative facts are established.

## Evidence board UI — `src/components/EvidenceBoard.tsx`

Full redesign. Current swim-lane layout replaced with an elimination workspace.

**Per checkpoint panel:**
- Checkpoint question as header
- One slot per wrong answer option (labelled with the answer text)
- Player drags a clue card from their collected clues into a wrong answer slot
- On drop: validate clue.contradicts includes this (checkpoint, answer) pair — show error
  state if not ("This clue doesn't contradict that option")
- When all slots are filled with valid clues: the remaining answer option highlights as
  proven and submission is automatic.
- For perpetrator and motive: panels show a locked state ("Establish the facts of the
  crime first") until all three investigative checkpoints are confirmed

**Clue cards:**
- No weight indicator (there is no weight)
- Show clue text
- Visual state: unassigned / assigned-to (shows which checkpoint/answer it's proving)

## `.claude/commands/review.md` changes

**Phase 2 audit replaces most existing checks with:**

### Coverage check
For every checkpoint, for every wrong answer, identify which clue(s) contradict it.
Build the full table. Any uncovered wrong answer = structural failure.

### Contradiction quality
For each (clue → wrong answer) pair: is the contradiction actually logically valid from
the clue text alone? "No puncture wound" validly contradicts "stabbing." "Witness seemed
nervous" does not validly contradict anything specific. Flag weak contradictions.

### Anti-clustering
Identify any clue with 3+ contradicts entries. Is it genuinely specific enough to
warrant that many contradictions, or is it a vague catch-all?

### Contradiction clarity
Ensure that no clue can be construed to contradict a correct option.

### Routing check
Do the clues that contradict perpetrator and motive wrong answers require deliberate
routing (item-chains, specific character visits)? Or are they available trivially from
the arrival location inspect?

### Arrival clue flood
Unchanged — still cap at 4 clues from the first inspect of arrival location.

**Remove:** convergence check, correct/red_herring coverage, locked-clue timing checks.
